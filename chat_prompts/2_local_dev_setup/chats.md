## Prompt 3

Can you go ahead with task 1.6:

* **Task 1.6:** Define a custom Docker bridge network in `docker-compose.yml` to allow the containers to communicate.

Here is the complete `docker-compose.yml` file. This fulfills both Task 1.5 and Task 1.6 by defining the three core services and explicitly connecting them via a custom bridge network.

This setup ensures your containers can securely resolve each other by their service names (e.g., the backend can reach the database at `db:5432`).

Create a file named `docker-compose.yml` at the root of your monorepo and add the following configuration:

```yaml

services:
  # ==========================================
  # Database Service (PostgreSQL)
  # ==========================================
  db:
    image: postgres:18.3-alpine3.23
    container_name: stock_db
    restart: always
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - app_network

  # ==========================================
  # Backend Service (FastAPI)
  # ==========================================
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile.dev
    container_name: stock_backend
    restart: unless-stopped
    ports:
      - "${BACKEND_PORT:-8000}:8000"
    environment:
      - DATABASE_URL=postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB}
      - GEMINI_API_KEY=${GEMINI_API_KEY}
    volumes:
      # Mount local backend directory to /app in container for hot-reloading
      - ./backend:/app
    depends_on:
      - db
    networks:
      - app_network

  # ==========================================
  # Frontend Service (React + Vite)
  # ==========================================
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile.dev
    container_name: stock_frontend
    restart: unless-stopped
    ports:
      - "${FRONTEND_PORT:-5173}:5173"
    volumes:
      # Mount local frontend directory to /app for HMR (Hot Module Replacement)
      - ./frontend:/app
      # Prevent local node_modules from overwriting container's node_modules
      - /app/node_modules
    environment:
      - VITE_API_URL=http://localhost:${BACKEND_PORT:-8000}
    depends_on:
      - backend
    networks:
      - app_network

# ==========================================
# Task 1.6: Custom Docker Bridge Network
# ==========================================
networks:
  app_network:
    driver: bridge
    name: stock_app_network

# ==========================================
# Persistent Volumes
# ==========================================
volumes:
  postgres_data:

```

### How This Achieves Your Goal

* **Networking (`app_network`):** By defining `app_network` as a `bridge` network at the bottom and attaching all three services to it, Docker's internal DNS allows the containers to discover each other. Notice how the backend's `DATABASE_URL` uses `@db:5432` instead of `localhost`.
* **Hot-Reloading (`volumes`):** The `./backend:/app` and `./frontend:/app` volume mounts map your local code directly into the running containers. When you save a file on your host machine, Uvicorn and Vite will instantly detect the change and restart/reload.
* **Environment Injection:** Compose automatically reads the `.env` file in the same directory and passes variables like `POSTGRES_USER` directly into the PostgreSQL and FastAPI containers.

3. **Frontend Port:** The production frontend exposes port `80` instead of `5173`, mimicking how standard web traffic will hit your Nginx server.

**`frontend/nginx.conf`**
This file tells the production Nginx server how to serve your React app, handle client-side routing, forward API requests, and upgrade the connection for your real-time Gemini chat WebSockets.

```nginx
server {
    listen 80;
    server_name localhost;

    # Serve the React application
    location / {
        root /usr/share/nginx/html;
        index index.html index.htm;
        # Required for React Router to handle client-side navigation
        try_files $uri $uri/ /index.html;
    }

    # Reverse Proxy for REST API requests
    location /api/ {
        proxy_pass http://backend:8000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Reverse Proxy for WebSocket connections (Feature 6)
    location /ws/ {
        proxy_pass http://backend:8000/ws/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_read_timeout 86400; # Keep WebSocket alive
    }
}

```

Implementing an application-level "Heartbeat" (Ping/Pong) ensures dead connections are pruned while active ones stay alive.

Here is how we update the Nginx configuration and implement the heartbeat mechanism across your frontend and backend.

### 2. The Client-Side Heartbeat (React / Zustand)

The easiest and most reliable way to handle this is to make the frontend the "pinger". When the WebSocket opens, we start a Javascript interval that sends a tiny JSON payload every 30 seconds. When the socket closes, we clear the interval.

Here is how you inject that into your `chatStore.ts`:

```typescript
  const socket = new WebSocket(`${import.meta.env.VITE_WS_URL}/ws/chat/${sessionId}`);

  socket.onopen = () => {
    console.log("WebSocket Connected");

    // START THE HEARTBEAT
    // Sends a ping every 30 seconds (30000 ms) to keep Nginx routing alive
    pingInterval = setInterval(() => {
      if (socket.readyState === WebSocket.OPEN) {
        socket.send(JSON.stringify({ type: "ping" }));
      }
    }, 30000);
  };

  socket.onmessage = (event) => {
    const data = JSON.parse(event.data);

    // Ignore pong responses; they just keep the connection alive
    if (data.type === "pong") return;

    // ... handle ai_chunk and ai_complete logic here ...
  };

  socket.onclose = () => {
    console.log("WebSocket Disconnected");
    // STOP THE HEARTBEAT to prevent memory leaks
    clearInterval(pingInterval);
  };
};

```

## Prompt 12

Using allow_methods=["*"] and allow_headers=["*"] are these security vulnerability, how to fix?

You have an excellent eye for security. Yes, using `allow_methods=["*"]` and `allow_headers=["*"]` is considered a security misconfiguration, especially in an enterprise-grade application, and *especially* when you are also using `allow_credentials=True`.

Here is a breakdown of why this is dangerous and exactly how we will fix it using our Object-Oriented constants.

### Why `["*"]` is a Security Risk

When you use the wildcard `*`, you are telling the browser: *"I trust the allowed origins to send absolutely any type of HTTP request with any custom header they want."*

1. **The Risk with `allow_methods=["*"]`:** If you only intend for an endpoint to accept `GET` and `POST` requests, but you allow `*`, an attacker who compromises an allowed origin (e.g., via a Cross-Site Scripting or XSS attack on your frontend) could forge `DELETE` or `PUT` requests to wipe out a user's portfolio or chat history.
2. **The Risk with `allow_headers=["*"]`:**
Modern web security relies heavily on custom headers (like CSRF tokens or specific authorization schemes). Allowing any header opens the door for attackers to probe your API with malformed or malicious headers, potentially bypassing web application firewalls (WAFs) or exploiting backend header-parsing vulnerabilities.
3. **The `allow_credentials` Multiplier:**
Because we must set `allow_credentials=True` to accept your secure `HttpOnly` JWT cookies, the stakes are much higher. If an attacker exploits the wildcards, they are doing so *fully authenticated* as the victim.

---

### How to Fix It (The Principle of Least Privilege)

To fix this, we apply the **Principle of Least Privilege**. We must explicitly define exactly which methods and headers our React frontend is allowed to use—and block everything else.

#### Step 1: Update `constants.py`

To keep our code clean and SOLID, we will define these allowed lists in our centralized constants file. Add this new class to your `/backend/constants.py`:

```python
class CORSConfig:
    """Explicitly defined CORS rules to prevent overly permissive wildcards."""

    # We allow OPTIONS because browsers use it for CORS "preflight" checks
    ALLOWED_METHODS = [
        "GET",
        "POST",
        "PUT",
        "DELETE",
        "OPTIONS"
    ]

    # Only allow standard safe headers and the specific headers React needs
    ALLOWED_HEADERS = [
        "Content-Type",
        "Authorization",
        "Accept",
        "Origin",
        "X-Requested-With"
    ]

```

#### Step 2: Update the App Factory in `main.py`

Now, we inject these strict constants into the CORS middleware configuration.

```python
# /backend/main.py (Snippet)
from constants import Environment, AppConfig, APITags, CORSConfig

class AppCreator:
    # ... (__init__ remains the same) ...

    def _configure_cors(self):

            f"http://localhost:{settings.frontend_port}",
            "http://127.0.0.1",
            "http://localhost"
        ]

        self.app.add_middleware(
            CORSMiddleware,
            allow_origins=origins,
            allow_credentials=True,

            # FIX: Replaced ["*"] with explicitly defined constants

```

By making this change, if a malicious script tries to send a weird HTTP method (like `TRACE` or `PATCH` if we don't use it) or a custom sneaky header, the browser and FastAPI will immediately block the request before it ever reaches your business logic.
